# Funding and Maintainer Support

If DevMaid is going to be distributed seriously, it helps to make support channels explicit from day one.

## Recommended support stack

Recurring support:

- primary landing page: `https://github.com/your-org/devmaid`
- GitHub Sponsors behind that landing page if enabled

One-time support:

- support page can route to Ko-fi or Buy Me a Coffee
- DMG, README, About, and Settings should all point to the same support URL

Professional support:

- company sponsorship for roadmap priorities
- paid implementation help for teams with custom toolchains
- release sponsorship for signed and notarized builds

## Publish-ready checklist

Primary public funding/support URL:

- `https://github.com/your-org/devmaid`
- company contact: `support@your-domain.example`

## Suggested README wording

Example:

> If DevMaid saves you hours of cleanup or helps recover disk space on your dev machine, you can support continued work through GitHub Sponsors or your public support page.

## Suggested sponsor tiers

- Supporter:
  helps cover build and release costs
- Power user:
  helps fund new cleanup categories and automation improvements
- Team sponsor:
  helps fund roadmap work, compatibility testing, and packaging polish

## What support can fund

- notarized releases
- CI and release automation
- wider coverage for dev tools and package managers
- SwiftUI GUI work
- documentation and recovery tooling
